## Phase 1 — MVP 개발 (M3~M8)

> 스프린트 단위(4주): 각 스프린트는 API 구현 → UI 구현 → 통합 테스트 순서로 진행

### Sprint 1 (M3): 프로젝트 기반 & 인증

**백엔드**

1. NestJS 프로젝트 초기화 (모듈 구조: auth / seniors / companies / matching / contracts / settlements)
2. TypeORM + PostgreSQL 연결, 마이그레이션 도구 설정
3. 데이터베이스 스키마 생성

   ```sql
   -- 핵심 6개 테이블
   users, senior_profiles, company_profiles,
   tf_requests, proposals, contracts, settlements, reviews
   ```

4. 인증 API 구현
   - `POST /auth/signup` — 역할(시니어/기업) 구분 가입
   - `POST /auth/login` — JWT Access Token + Refresh Token 발급
   - bcrypt 비밀번호 해싱, HTTPS 설정

5. RBAC 미들웨어 구현 (`senior` / `company` / `admin` 역할)

**프론트엔드**

6. React Native + Expo 프로젝트 초기화 (iOS/Android 공통)
7. Next.js 프로젝트 초기화 (기업용 웹 대시보드)
8. 공통 디자인 토큰 적용 (DESIGN.md 색상·타이포그래피 시스템)
9. 로그인·회원가입 화면 구현 (역할 선택 → 정보 입력)

**인프라**

10. VPS 서버 구성 (Ubuntu 22.04, Docker + Nginx + Let's Encrypt SSL)
11. GitHub Actions 배포 파이프라인 연결 (`develop` 브랜치 → VPS 자동 배포)

**Sprint 1 완료 기준**

- [ ] `/auth/signup`, `/auth/login` API 정상 동작
- [ ] JWT 토큰 발급 및 RBAC 보호 라우트 검증
- [ ] 모바일·웹 로그인 화면에서 실제 API 연동 성공

### Sprint 2 (M4): 프로필 & TF 요청

**백엔드**

1. 시니어 프로필 API
   - `POST /seniors/profile` — 전문 분야, 경력, 기술 키워드, 가용 기간, 지역 등록
   - `GET /seniors/profile/:id` — 프로필 조회
   - `PUT /seniors/profile/:id` — 프로필 수정
   - 프로필 공개 여부(`status`) 토글

2. 기업 TF 요청 API
   - `POST /requests` — 프로젝트명, 필요 분야, 기간, 예산, 목표 등록
   - `GET /requests/:id` — 요청 조회
   - `GET /requests` — 기업별 요청 목록

3. AWS S3 / MinIO 파일 저장소 연동 (프로필 이미지 업로드)

**프론트엔드**

4. 시니어 프로필 등록 화면 (단계별 입력: 전문 분야 → 경력 → 기술 태그 → 가용 기간)
5. 기업 TF 요청 작성 화면 (드롭다운 + 텍스트 입력)
6. 경력 태그 칩 컴포넌트 구현 (DESIGN.md 칩 스펙 적용)

**접근성 체크**

- 모든 입력 필드: 레이블이 플레이스홀더와 분리되어 항상 표시
- 버튼 최소 높이 56px, 터치 영역 44px 이상
- 폰트 크기 최소 16px 강제

**Sprint 2 완료 기준**

### Sprint 3 (M5): AI 매칭 엔진 v1

**매칭 알고리즘 구현 (룰 기반)**

1. 필터 단계 구현

   ```typescript
   // 후보군 축소 조건
   filter(senior):
     isWithinBudget(senior, request)      // 예산 범위
     && isAvailable(senior, request)      // 가용 기간 겹침
     && isInRegion(senior, request)       // 지역 일치
     && hasRequiredField(senior, request) // 필요 분야 포함
   ```

2. 점수화 단계 구현

   | 항목 | 가중치 |
   | ---- | ------ |
   | 전문 분야 일치도 | 30% |
   | 경력 적합도 (연차·직급) | 25% |
   | 기간·지역 일치도 | 25% |
   | 과거 프로젝트 성과 (리뷰 평점) | 20% |

3. 매칭 API 구현
   - `GET /requests/:id/matches` — 적합도 점수 내림차순 추천 리스트 반환
   - 응답에 `fit_score`, `match_reasons[]` 포함

4. Redis 캐싱 적용 (동일 요청 반복 조회 최적화, TTL 10분)

**프론트엔드**

5. 매칭 결과 화면 (추천 시니어 카드 리스트, 적합도 점수 시각화)
6. 시니어 홈 화면 (수신된 매칭 제안 카드)

**Sprint 3 완료 기준**

- [ ] 더미 시니어 데이터 20건으로 매칭 결과 도출 확인
- [ ] 필터·점수 로직 단위 테스트 (Jest) 커버리지 80% 이상
- [ ] 매칭 API 응답 시간 500ms 이내

### Sprint 4 (M6): 제안 플로우

**백엔드**

1. 제안 API 구현
   - `POST /requests/:id/proposals` — 기업 → 시니어 제안 발송
   - `GET /proposals/:id` — 제안 상세 조회
   - `POST /proposals/:id/accept` — 시니어 수락 (→ 계약 생성 트리거)
   - `POST /proposals/:id/reject` — 시니어 거절
2. 푸시 알림 연동 (FCM) — 제안 수신 시 시니어에게 알림

**프론트엔드**

3. 기업: 제안 발송 화면 (추천 전문가 카드 → 제안 메시지 입력 → 발송)
4. 시니어: 제안 수신함 화면 (프로젝트 세부 사항 + 수락/거절 버튼)
5. 제안 상태 배지 컴포넌트 (대기 / 수락 / 거절)

**Sprint 4 완료 기준**

- [ ] 제안 발송 → 시니어 앱 푸시 알림 수신 확인
- [ ] 수락/거절 상태 변경 및 UI 반영 확인

### Sprint 5 (M7): 계약 자동 생성 & 채팅

**백엔드**

1. 계약 자동 생성 API
   - `POST /contracts` — 제안 수락 시 자동 호출
   - HTML 계약서 템플릿 → Puppeteer PDF 변환 → S3 저장
   - 계약 항목: 프로젝트 기간, 역할 범위, 보수, 정산 방식

2. 계약 조회 API
   - `GET /contracts/:id` — 계약서 PDF 링크 포함 반환

3. 채팅 기능 구현 (Socket.IO)
   - 계약 체결 후 기업·시니어 간 1:1 채팅방 자동 생성
   - Redis Pub/Sub으로 실시간 메시지 처리

**프론트엔드**

4. 계약서 확인 화면 (PDF 미리보기, 수락 확인 버튼)
5. 채팅 화면 (메시지 목록, 입력창)

**Sprint 5 완료 기준**

### Sprint 6 (M8): 정산·진행 관리 & QA

**백엔드**

1. 정산 API 구현 (Toss Payments 에스크로 연동)
   - `POST /contracts/:id/settlement` — 정산 요청
   - `GET /contracts/:id/status` — 진행 상태 조회 (`진행 중` / `정산 요청` / `완료`)
   - 에스크로 플로우: 기업 선결제 → 플랫폼 보관 → 완료 후 시니어 지급

2. 리뷰 API
   - `POST /contracts/:id/review` — 완료 후 양측 평가 (1~5점 + 코멘트)

3. 진행률 업데이트 API
   - `PUT /contracts/:id/progress` — 0~100% 진행률 기록

**프론트엔드**

4. 진행 관리 화면 (진행률 바, 정산 요청 버튼)
5. 완료 후 리뷰 입력 화면 (별점 + 코멘트)
6. 기업 대시보드 (진행 중 프로젝트 카드, 정산 내역)

**QA & 배포**

7. 통합 테스트 (Supertest) — 핵심 플로우 전체 검증
   - 가입 → 프로필 → TF 요청 → 매칭 → 제안 → 계약 → 정산
8. 시니어 접근성 검수 (글씨 크기, 터치 영역, 색상 대비)
9. VPS 운영 환경 배포 및 모니터링 구성 (Prometheus + Grafana)

- [ ] 전체 플로우 E2E 테스트 통과
- [ ] Toss 에스크로 테스트 결제 정상 처리

## 개발 우선순위 원칙

1. **매칭이 작동해야 가치가 증명된다** — 스프린트 3(매칭 엔진)은 절대 지연 불가
2. **시니어 접근성은 기능과 동등한 우선순위** — 각 스프린트 완료 전 접근성 체크 필수
3. **초기에는 수동 큐레이션 병행** — AI 매칭 정확도가 낮은 경우 운영자가 보완 추천
4. **VPS 먼저, AWS는 정식 출시 후** — Phase 1~2는 VPS(월 25만원)로 운영, Phase 3에서 전환
